Целевые сценарии использования платформы. Упорядочены по сложности — простые блокируют сложные.

## Dependency Graph

```
US0 (базовый пайплайн)
 ├── US1 (свой токен) → US2 (авто-provisioning)

```

## US1: Свой токен Telegram бота
**Приоритет**: Критический | **Статус**: Done | **Blocked by**: —

**Как** пользователь
**Хочу** использовать свой Telegram Bot Token
**Чтобы** бот имел моё имя и был под моим контролем

**Сценарий**:
1. Система спрашивает: "Хотите создать токен сами?"
2. Пользователь регистрирует бота через @BotFather
3. Пользователь скидывает токен
4. Система деплоит бот с этим токеном

**Acceptance Criteria**:
- [x] PO принимает токен от пользователя через Telegram → 2026-02-15
- [x] Токен сохраняется как секрет проекта (Fernet) → 2026-02-25
- [x] DevOps subgraph инжектит токен в deploy → 2026-02-15
- [x] E2E сценарий с tg_bot модулем проходит → weather_bot PASS 2026-03-04

**E2E**: weather_bot PASS 2026-03-04 (15 min, token injection + tg_bot polling verified)

## US2: Создание бота по запросу (Auto-provisioning)
**Приоритет**: Высокий | **Статус**: Not started | **Blocked by**: US1

**Как** пользователь
**Хочу** описать словами какой бот мне нужен
**Чтобы** получить работающий бот через 10-15 минут (без регистрации в BotFather)

**Сценарий**:
1. Пользователь пишет: "Сделай бота погоды"
2. Система выделяет токен из пула (или регистрирует сама)
3. Система генерирует код, тестирует, деплоит
4. Пользователь получает ссылку на готового бота

**Acceptance Criteria**:
- [ ] Пул свободных токенов → Roadmap Phase 4 (idea: Telegram Bot Pool)
- [ ] Проект создаётся автоматически с выделенным токеном → Roadmap Phase 4
- [ ] Пользователь не взаимодействует с BotFather → Roadmap Phase 4

**E2E**: — | Roadmap Phase 4
