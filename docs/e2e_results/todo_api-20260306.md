### Problem 2: No router generation from spec
- **Type**: template
- **Severity**: minor
- **Backlog**: `template (triaged)` — duplicate, tracked in service-template backlog ("Auto-generate routers from domain specs")
- **Description**: The framework generates protocols and controller stubs from spec, but NOT routers. Since the spec already contains HTTP method, path, status codes, and params, router generation would eliminate ~100 lines of boilerplate per domain.
- **Root cause**: Router generation not yet implemented in framework.
- **Suggested fix**: Add router generation to `make generate-from-spec` pipeline.

### Problem 3: Inconsistent import style in generated controller stubs
- **Type**: template
- **Severity**: minor
- **Backlog**: `template (triaged)` — duplicate, tracked in service-template backlog (eager import chain / lazy imports)
- **Description**: Generated controller stubs use relative imports (`from ..generated.protocols import ...`) while existing controllers use absolute imports (`from services.backend.src.generated.protocols import ...`).
- **Root cause**: Controller stub generator uses relative imports by default.
- **Suggested fix**: Align generated imports with project convention (absolute).

### Problem 4: No documentation on ORM base class choices
- **Type**: template
- **Severity**: minor
- **Backlog**: `template (triaged)` — duplicate, already fixed (CreatedAtMixin added in `b0afb10`), docs tracked in service-template backlog
- **Description**: No guidance on when to use `ORMBase` (has created_at + updated_at) vs `CreatedAtMixin + Base` (created_at only). Worker had to figure this out independently.
