## Problems Found

### Problem 1: Broken import in scaffolded user repository
- **Type**: template
- **Severity**: major
- **Backlog**: service-template backlog
- **Description**: `services/backend/src/app/repositories/user.py:9` imports `from ..schemas import UserCreate, UserUpdate` but the schemas module doesn't exist at that path — schemas are generated in `shared/shared/generated/schemas.py`. This causes `ModuleNotFoundError` and blocks `make migrate`, `make tests`, and any backend import.
- **Root cause**: The scaffold generator produces incorrect import paths for schemas in repository files. It references a relative `schemas` module in `app/` instead of the actual generated location.
- **Suggested fix**: Update the scaffold template to generate `from shared.generated.schemas import UserCreate, UserUpdate` or add a re-export module at `services/backend/src/app/schemas.py`.

### Problem 2: Eager import chains cause fragility
- **Type**: template
- **Severity**: minor
- **Backlog**: service-template backlog
- **Description**: `services/backend/src/__init__.py` eagerly imports `app` and `create_app`, and `api/__init__.py` eagerly imports the router. This means any broken import in the chain (routers → controllers → repositories → schemas) causes the entire application to fail at import time. Alembic migrations are also affected since `env.py` triggers full app initialization.
- **Root cause**: Python's import side-effects combined with eagerly importing the full app tree in `__init__.py` files.
- **Suggested fix**: Use lazy imports or have Alembic `env.py` import models directly without going through the app initialization chain.

### Problem 4: Deploy directory uses hyphenated name, not underscored
- **Type**: meta
- **Severity**: minor
- **Backlog**: #35
- **Description**: The skill instructions use `$PROJECT_NAME` (todo_api) for server paths, but the actual deploy directory is `todo-api` (hyphenated). The `ssh root@$SERVER_IP "cd /opt/services/$PROJECT_NAME/infra"` command failed because of this mismatch.
