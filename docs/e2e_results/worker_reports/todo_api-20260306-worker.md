# Audit Report

## Overview
Audit of the service-template framework while implementing a TODO CRUD API.

## What Worked Well

1. **Spec-first workflow**: Defining models in `shared/spec/models.yaml` and domain operations in `services/backend/spec/todos.yaml`, then running `make generate-from-spec` produced correct Pydantic schemas, protocols, and controller stubs. Very smooth.

2. **Code generation quality**: Generated schemas correctly handled `readonly` fields (excluded from Create variants), `optional` fields in Update variants, and default values. The protocols matched the spec exactly.

3. **Controller stub generation**: The framework generated a controller stub with `NotImplementedError` placeholders for each operation. This is a good starting point - clear what needs to be implemented.

4. **Database migration flow**: `orchestrator dev-env start-infra db` + `make migrate` + `make makemigrations` + `make migrate` worked without issues. The autogenerated migration correctly detected the new `todos` table.

5. **Linting and spec compliance**: `make lint` caught an import ordering issue immediately, and `make format` fixed it. The spec compliance and controller sync checks all passed on first try.

6. **Test infrastructure**: The conftest.py with SQLite-backed transactional fixtures works well for unit tests. Test isolation via savepoints is properly configured.

## Issues Encountered

1. **Import ordering in generated controller stub**: The generated `services/backend/src/controllers/todos.py` uses relative imports (`from ..generated.protocols import TodosControllerProtocol`) while the existing user controller uses absolute imports (`from services.backend.src.generated.protocols import UsersControllerProtocol`). This inconsistency is minor but could confuse developers. I rewrote the controller with absolute imports to match the user controller pattern.

2. **No router generation**: The framework generates protocols and controller stubs, but does NOT generate routers. The AGENTS.md says "Routers are written manually" which is fine, but since the spec already contains all the information needed (HTTP method, path, status codes, params), router generation would save significant boilerplate. Each router file is ~100 lines of very predictable code.

3. **AGENTS.md is in Russian**: The `AGENTS.md` and `services/backend/AGENTS.md` files are written in Russian. This could be a barrier for non-Russian-speaking developers. Consider either providing English versions or making the language configurable in the template.

4. **Todo model doesn't need updated_at**: The spec defines `created_at` but not `updated_at` for Todo. The framework provides `ORMBase` (with both created_at + updated_at) and `CreatedAtMixin` (created_at only). I used `CreatedAtMixin + Base` directly. This works but isn't documented anywhere - the AGENTS.md only shows `ORMBase` usage.

5. **No guidance on partial-update base classes**: The `CreatedAtMixin` and `ORMBase` are useful, but there's no documentation about which to use when. A brief note in AGENTS.md about these base classes would help.

## Suggestions for Improvement

1. **Generate routers automatically**: Since the domain spec already has all REST metadata (method, path, status, params), router generation would eliminate ~100 lines of boilerplate per domain and reduce the chance of spec-router drift.

2. **Make controller stub imports consistent**: Use either relative or absolute imports in generated stubs - currently the generated stub uses relative imports which differ from the existing codebase pattern.

3. **Add English translations or make language configurable**: Documentation files should be accessible to all developers.

4. **Document ORM base class choices**: Add a brief section to AGENTS.md explaining when to use `ORMBase` vs `CreatedAtMixin + Base` vs plain `Base`.

5. **Consider a `make new-domain` command**: A single command that creates the spec YAML, generates code, and scaffolds the router would streamline adding new domains.
