## Problems Found

### Problem 1: Scaffold skipped because repo already existed

- **Type**: orchestrator
- **Severity**: critical
- **Description**: The `todo-api` GitHub repo existed from a previous test run. The engineering worker logged `repo_already_exists` and skipped repo creation. The worker-manager cloned the existing repo (which had no scaffold content from the template) instead of running the copier scaffold phase.
- **Root cause**: When the repo already exists, the worker-manager does `git_repo_setup` (clone) instead of `scaffold_phase` (copier). The previous run apparently created the repo but either didn't scaffold properly, or the repo was left over from a failed run.
- **Impact**: No `.github/workflows/ci.yml`, no `.framework/`, no template structure. The Claude agent had to implement everything from scratch without the scaffolded foundation.
- **Suggested fix**: Either (a) the E2E test should ensure the repo is deleted before starting, or (b) the engineering worker should detect that the existing repo is missing scaffold markers (e.g., `.copier-answers.yml`) and re-scaffold.

### Problem 2: CI gate fails immediately with WorkflowNotFoundError

- **Type**: orchestrator
- **Severity**: major
- **Description**: The CI gate checked for `ci.yml` workflow runs and got a 404. The fail-fast behavior (`CI failed after 1 attempt(s), retries exhausted`) is correct given the recent `WorkflowNotFoundError` fix, but the root issue is that the scaffold was missing.
- **Root cause**: Direct consequence of Problem 1 — no `.github/workflows/ci.yml` in the repo.
- **Impact**: Task failed even though the code generation itself succeeded.

### Problem 3: Worker environment issues (from worker audit)

- **Type**: template
- **Severity**: major
- **Description**: Multiple Python environment issues in the worker container:
  1. `make install` fails — assumes virtualenv exists but none is pre-created
  2. `pip install` fails with permission denied (non-root, no write to site-packages)
  3. User site-packages not on `sys.path` — packages installed with `--user` not importable
  4. Makefile targets don't account for user-installed packages
- **Root cause**: Worker container Python environment is not properly configured for the non-root `worker` user.
- **Suggested fix**: Either pre-create a virtualenv in the worker image, or ensure user site-packages are on the default Python path.

### Problem 4: Missing AGENTS.md and non-functional code generation

- **Type**: template
- **Severity**: minor
- **Description**: TASK.md references `AGENTS.md` for code patterns (doesn't exist) and `make generate-from-spec` (not implemented — just prints a message). The Claude agent had to infer all patterns.
- **Root cause**: Template doesn't include AGENTS.md. Code generation from spec files is not wired up.
- **Suggested fix**: Either include AGENTS.md in the scaffold or remove references. Implement or remove `generate-from-spec`.

## Summary

The code generation itself worked well — the Claude agent produced a functional Todo API in ~5 minutes with proper layered architecture, tests, and an excellent audit report. However, the test failed at the CI gate because the scaffold phase was skipped (repo already existed without template content). This is a test environment issue, not a fundamental orchestrator bug, though the orchestrator could be more resilient by detecting missing scaffold markers.
