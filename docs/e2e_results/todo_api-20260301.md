## Problems Found

### Problem 1: Stale in-memory `project` dict causes scaffold skip (CRITICAL)

- **Type**: orchestrator
- **Severity**: critical
- **Description**: Scaffold phase is silently skipped because the developer node receives `project_spec` with `status="draft"` instead of `"scaffolding"`. The `_build_scaffold_config()` check on `developer.py:227` rejects it.
- **Root cause**: In `engineering_worker.py`, the flow is:
  1. **Line 518**: `project = await api_client.get_project(project_id)` — fetches project with `status="draft"`
  2. **Line 562**: `_create_repo_and_set_secrets(project)` — updates **DB** status to `"scaffolding"` (line 460)
  3. **Line 628**: `subgraph_input` uses `project_spec=project` — **still the stale dict with `status="draft"`**
  4. Developer node reads `project_spec` from state → sees `"draft"` → `_build_scaffold_config` returns `None`

  The previous investigation (Bug 1 in `docs/investigations/e2e-line2-scaffold-skip-bug.md`) fixed the premature `"developing"` status update but missed this: the in-memory `project` dict is **never refreshed** after `_create_repo_and_set_secrets` updates the DB.
- **Suggested fix**: After `_create_repo_and_set_secrets(project)` at line 562, update the in-memory dict:
  ```python
  project["status"] = ProjectStatus.SCAFFOLDING.value
  ```
  Or re-fetch: `project = await api_client.get_project(project_id)`

### Problem 2: Worker environment not set up for development

- **Type**: template
- **Severity**: major
- **Description**: Without scaffold, the worker has no virtualenv, no pre-installed dependencies, and pip install fails with permission denied on system site-packages. The agent resorted to `pip install --user` and `PYTHONPATH` hacks. This is the same issue as the previous run's Problem 3.
- **Root cause**: Worker container's Python environment assumes scaffolded project with `make install` creating a venv. Without scaffold, the raw environment is hostile to development.

### Problem 3: Task prompt says "already scaffolded" when it wasn't

- **Type**: orchestrator
- **Severity**: minor
- **Description**: The task prompt sent to Claude Code says "The project was scaffolded with copier from service-template" and lists directories that don't exist. This misleads the agent into searching for non-existent files.
- **Root cause**: `_build_task_message()` in `developer.py` always includes the "already scaffolded" section regardless of whether scaffold actually ran.
- **Suggested fix**: Make the scaffolded section conditional on `scaffold_config is not None`.

## Root Cause Analysis

This is the **same fundamental bug** as documented in `docs/investigations/e2e-line2-scaffold-skip-bug.md` but with a subtly different trigger. The investigation found and fixed the premature `"developing"` status update. However, the fix introduced a new variant: the `project` dict fetched at line 518 (with `status="draft"`) is passed through to the subgraph without being refreshed after the status changes to `"scaffolding"` at line 460.

The fix is a one-line change in `engineering_worker.py` after `_create_repo_and_set_secrets(project)`.

## Summary

The scaffold phase was completely skipped due to a stale in-memory project status. The Claude agent received an empty repo and built a functional Todo API from scratch in ~7 minutes (demonstrating the agent works well), but without `.github/workflows/ci.yml` from the template, the CI gate failed immediately. This is a critical orchestrator bug blocking all Level B/C E2E tests.
