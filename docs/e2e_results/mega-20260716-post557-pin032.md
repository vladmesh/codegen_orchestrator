## Cleanup

Первый `make test-live-clean` упал на fence отмены активных runs: `GET /api/runs/?project_id=...`
отвечал 500 на все три манифеста (сегодняшний + два stale). Два дефекта:

1. В `list_runs` и `list_projects` query-параметр `status` затенял модуль `fastapi.status`:
   ветка 401 для неаутентифицированного вызова падала `AttributeError` → 500.
2. Manifest-recovery в `scripts/clean_live_tests.py` строил httpx-клиент вообще без auth,
   т.е. fence и не мог пройти (то же семейство, что #73/#75).

Оба починены в PR #81 (main `e428435d`): параметры переименованы с `Query(alias="status")`
(публичный контракт не менялся), клиент recovery шлёт `X-Internal-Key`, добавлен regression-тест
на 401. После пересборки `api` повторный `make test-live-clean` прошёл (exit 0), манифесты
удалены, DB/GitHub/workspaces чисты.

**Найден ещё один дефект, пока не исправлен**: серверная чистка гасит и верифицирует compose-проект
по имени из манифеста `live-test-2c3e830f` (дефисы), а реально контейнеры бежали под проектом
`live_test_2c3e830f` (подчёркивания — slug из шаблона, top-level `name:` в generated compose).
Фильтр `label=com.docker.compose.project=live-test-...` не находит ничего, `docker compose -p` бьёт
мимо, а финальный `test -z` проходит по той же причине. Итог: каталог `/opt/services/...` удалён,
а контейнеры остались Up. Убраны вручную (containers + volume + network + image), идея заведена
на доску. Верификация чистки не должна использовать то же потенциально неверное имя, что и
удаление: каноничное имя проекта надо брать из фактического label контейнеров либо
зафиксировать канон deploy↔harness.

Косметика: в registry-каталоге копятся пустые имена `live-test-*` (17 шт., `tags: null`) —
данные удалены, имена уйдут только после registry garbage-collect.
