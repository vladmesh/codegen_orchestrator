# QA Node Brainstorm

## Идея

Добавить QA-ноду в граф после деплоя. Не юнит/интеграционные тесты (это ответственность разработчика), а полноценное ручное тестирование — бот тыкает кнопки, ходит по страницам, проверяет поведение как реальный пользователь.

## Что решили

### Flow

```
PO (спека + acceptance criteria)
  → Engineering (код + unit tests)
    → Deploy
      → QA Node:
          вход: acceptance criteria + URL/bot handle + creds
          1. читает acceptance criteria из спеки
          2. генерит конкретные test steps
          3. выполняет (Telethon / Playwright / httpx)
          4. репортит: pass/fail + детали
            → pass → notify user
            → fail → назад в Engineering с описанием бага
```

### Инструменты

| Что тестируем | Инструмент | Почему |
|---|---|---|
| Telegram-бот (команды, кнопки, диалоги) | Telethon | API-клиент, эмулирует юзера, браузер не нужен |
| Веб-интерфейс (сайт, админка) | Playwright MCP | Браузерная автоматизация |
| Telegram Mini Apps | Playwright MCP | WebView = браузер |
| REST API | httpx / aiohttp | HTTP-запросы |

### Кто генерит ожидаемое поведение

PO пишет **acceptance criteria** в спеке — высокоуровневые сценарии на человеческом языке. Не тест-кейсы, а "что должно работать":

```
## Acceptance Criteria
- /start → бот отвечает приветствием и показывает главное меню (3 кнопки)
- Кнопка "Создать заказ" → бот просит ввести название
- GET /health → 200
```

QA-нода **разворачивает** эти критерии в конкретные шаги и выполняет.

Почему не разработчик: он тестирует свои предположения, а не требования.
Почему не отдельная QA Planner нода: оверинжиниринг на старте.

### Среда тестирования

На старте — **тестируем на проде** с отдельным Telegram-аккаунтом. Причины:
- Минимум инфраструктурных изменений
- Не нужно учить service-template деплоить в два режима
- Бот один, продовый

Telegram Test Environment (test.telegram.org, DC2) отложен — потребует поддержки `TELEGRAM_TEST_MODE` в service-template и деплой бота дважды.

### Telegram Test Environment (справка)

- Полностью изолированная среда Telegram, бесплатная
- Тестовые номера: `99966XYYYY` (X = DC 1-3), код подтверждения = `XXXXX` (например DC2 → `22222`)
- Реальный номер не нужен, но аккаунт может использовать кто угодно (код известен)
- Отдельный BotFather, отдельные боты, отдельные токены
- Telethon/Pyrogram поддерживают `test_mode=True`

### Архитектура QA-воркера

```
QA Worker Container:
  - Claude (мозг, принимает решения)
  - Telethon (Telegram userbot)
  - Playwright (браузер, headless Chromium)
  - httpx (API-запросы)
```

Playwright и Telethon подключаются как MCP-серверы либо как нативные tools, Claude управляет ими.

## Открытые вопросы

### 1. Формат acceptance criteria
Нужно формализовать формат, чтобы QA-нода могла надёжно их парсить. Свободный текст или структурированный YAML? Насколько детальными должны быть критерии? PO — LLM, ему нужен чёткий шаблон.

### 2. Cleanup на проде
Если тестируем на проде — QA-нода создаёт записи в БД, отправляет сообщения. Нужен механизм cleanup:
- Отдельный test tenant / test user в БД?
- Пометка тестовых данных для последующей чистки?
- Или просто забить и чистить вручную?

### 3. Отдельный Telegram-аккаунт для QA
Нужен реальный номер телефона для аккаунта на проде. Варианты:
- Виртуальный номер (Twilio и т.д.) — риск бана со стороны Telegram
- Отдельная SIM-карта — надёжнее, но физический объект
- Как хранить сессию Telethon? В secrets проекта?

### 4. Retry и flaky-результаты
AI-driven тесты недетерминированы. Claude может интерпретировать ответ бота по-разному. Нужна стратегия:
- Сколько retry на один сценарий?
- Как отличить баг от flaky-теста?
- Жёсткие assertions vs мягкие (contains vs exact match)?

### 5. Scope тестирования
- Smoke (критический путь) на каждый деплой vs full regression по запросу?
- Кто определяет scope — PO, пользователь, автоматически?

### 6. Как QA-нода репортит баги обратно в Engineering
- Формат баг-репорта (что сделал → что ожидал → что получил)
- Сколько итераций fix loop до эскалации человеку?
- Скриншоты, логи — что прикладывать?

### 7. Стоимость и время
- Полный QA-прогон через Claude = минуты + доллары за токены
- Playwright скриншоты в контексте — дорого
- Нужно ли ограничивать бюджет на QA-ноду?

### 8. Staging environment (фаза 2)
Когда дойдём до staging:
- Отдельный docker-compose с тестовыми env-ами?
- Поднимается автоматически перед QA, сносится после?
- Нужна поддержка в service-template (`docker-compose.test.yml`)
- Telegram Test Environment подключать на этом этапе?

## Фазы реализации (примерный план)

1. Формализовать acceptance criteria в спеке PO
2. QA-нода: Telethon + Claude, тестирование Telegram-ботов на проде
3. Добавить Playwright для веб-интерфейсов
4. Staging environment
5. Visual regression, полный staging сервер
