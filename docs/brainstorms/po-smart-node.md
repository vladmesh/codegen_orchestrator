# Brainstorm: Smart PO Node

> **Дата**: 2026-03-06
> **Контекст**: Подготовка к запуску MVP. Цель: сделать PO более "умным" и полезным для нетехнических пользователей с минимальными усилиями.
> **Status**: triaged

---

## Current State
Текущая реализация PO (Product Owner) работает преимущественно как прямолинейный маршрутизатор:
1. Запрашивает токен Telegram (если это бот).
2. Создает проект.
3. Сразу вызывает `trigger_engineering`, передавая изначальный запрос пользователя (зачастую состоящий из пары слов) напрямую инженерам. 

## Problem / Opportunity
Нетехнические пользователи часто формулируют задачи слишком абстрактно (например, "Хочу бота для продажи цветов"). Изначального описания недостаточно для написания качественного кода. 
Возникает **возможность (Opportunity)**: научить PO общаться с пользователем (сократический диалог), задавать уточняющие вопросы и формировать детализированное техническое задание **до** передачи задачи инженерам, а также (исходя из обсуждения) дать PO возможность искать информацию в интернете (например, документацию по API).

## Options

### Option A: Prompt Engineering (Сократический диалог)
Изменение системного промпта PO (`services/langgraph/src/po/prompts.py`) без изменения логики графа. PO будет проинструктирован задавать уточняющие вопросы (2-3 вопроса) при получении абстрактных запросов и агрегировать ответы в большое ТЗ перед вызовом `trigger_engineering`.
- **(+)** Очень быстро (MVP ready, 15 минут).
- **(+)** Не требует написания нового кода или изменения графа.
- **(+)** Существенно улучшает качество генерируемого кода (через выдачу качественного промпта инженерам).
- **(-)** Полностью полагается на возможности LLM следовать инструкциям промпта.

### Option B: Добавление инструмента Web-поиска (DuckDuckGo Search)
Добавить инструмент веб-поиска (например, DuckDuckGo) в `services/langgraph/src/po/tools.py`. Инструмент будет вызываться PO, если необходимо найти документацию по сторонним API или примеры реализации перед написанием ТЗ.
- **(+)** Делает PO гораздо более автономным и "умным" (Пост-MVP или даже сейчас, так как это просто).
- **(+)** Повышает вероятность успешной интеграции сторонних сервисов.
- **(-)** Чуть-чуть увеличивает задержку перед ответом, так как добавляется вызов инструмента поиска.
- **(-)** Может излишне усложнить флоу, если PO начнет искать то, что не нужно.

### Option C: Полноценный этап Requirement Gathering (Graph Node)
Выделение отдельной ноды в LangGraph специально для сбора требований.
- **(+)** Максимальный контроль над процессом.
- **(-)** Слишком долго для "наименьшей крови" перед стартом MVP.

## Recommendation
Комбинация **Option A** и **Option B**. 
* Обновляем системный промпт для включения Сократического диалога: учим PO не спешить с вызовом инженеров, а сначала собирать требования, валидировать их, и только собрав детальное ТЗ, пушить его в `description`.
* Добавляем простую функцию поиска `duckduckgo_search` в арсенал тулзов PO, и прописываем в промпте правило: "Используй поиск, если пользователь просит прикрутить стороннее API, чтобы добавить документацию в ТЗ".
