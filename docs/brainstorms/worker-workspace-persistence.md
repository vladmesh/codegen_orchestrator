# Worker Workspace Persistence

## Problem

При таймауте или крэше воркера весь написанный код теряется.

### Наблюдение из e2e (2026-02-20)

1. PO получил задачу, создал проект `reverse-message-bot`, тригернул engineering task
2. Engineering-worker заспавнил воркер `dev-reverse-message-bot-24d2e336`
3. Claude Code внутри воркера за 10 минут написал 13 файлов (+176 строк): модели, контроллеры, роутеры, миграции, тесты, tg_bot handler
4. Claude Code **не успел закоммитить** до таймаута (600s)
5. Wrapper убил процесс, spawner отправил `DeleteWorkerCommand`
6. `delete_worker()` удалил контейнер + workspace — **весь код потерян**
7. PO увидел failure, сразу тригернул новый engineering task
8. Новый воркер клонировал репо с GitHub — только scaffold-коммит, все наработки отсутствуют
9. Claude Code начал всё заново — тройной расход токенов (третья попытка тоже не успела)

### Корневые причины

- **Workspace привязан к worker_id**: каждый воркер получает уникальный `{worker_id}`, при удалении workspace стирается
- **Нет сохранения прогресса**: Claude Code не коммитит промежуточные результаты, нет инструкции это делать
- **Нет передачи контекста между попытками**: новый воркер не знает что делал предыдущий
- **PO бесконтрольно ретрайит**: нет лимита на повторные попытки

### Дополнительный баг (пофикшен)

Worker-wrapper при ошибке не публиковал результат в output stream. Spawner зависал на 30 мин вместо того чтобы сразу вернуть ошибку.

**Фикс**: wrapper теперь публикует `{"status": "failed", "error": "..."}` в output stream при любой ошибке (коммит `dev-env-architecture`).

## Варианты решения

### Вариант A: Push WIP в git перед kill

Перед убийством процесса — `git add -A && git commit -m "WIP" && git push`.

**Плюсы:**
- Простая реализация (3-5 строк в wrapper timeout handler)
- Работа сохраняется в git — новый воркер клонирует и видит код

**Минусы:**
- Код в середине разработки: не линтован, возможно не компилируется
- Push в обход git hooks — CI упадёт на WIP-коммите, создаст шум
- Засоряет git историю WIP-коммитами
- Нужно решать: пушить в main или в отдельную ветку?

### Вариант B: Workspace по project_id + PROGRESS.md (рекомендуемый)

Workspace привязан к проекту, а не к воркеру. Claude Code ведёт PROGRESS.md с планом работы.

**Изменения:**

1. **Workspace path**: `/tmp/codegen/workspaces/{project_id}/` вместо `/{worker_id}/`
2. **create_worker**: если workspace для project_id уже существует — не клонировать, монтировать как есть
3. **delete_worker**: удалять контейнер, сеть, Redis keys — но **не workspace**
4. **GC workspace'ов**: отдельная джоба, чистит по возрасту или статусу проекта (completed/deployed)
5. **PROGRESS.md**: инструкция в CLAUDE.md — "создай план в PROGRESS.md, отмечай выполненные шаги"
6. **Два режима промпта**:
   - Initial: "Read TASK.md, create PROGRESS.md, implement"
   - Resume: "Previous developer didn't finish. Check PROGRESS.md and code state. Continue."

**Плюсы:**
- Код не теряется при крэше/таймауте/OOM/перезагрузке
- Новый воркер видит всё что было сделано — не начинает заново
- PROGRESS.md даёт контекст и наследнику, и нам для отладки
- Не засоряет git — ничего не пушится до готовности
- Git clone overhead для маленьких проектов мизерный (1-2 сек), но при resume его нет вообще

**Минусы:**
- Dirty state в workspace — но это решается промптом ("review what's here")
- Нужен GC для старых workspace'ов
- Slightly more complex logic в worker-manager

**Concurrency**: не проблема — один воркер на проект одновременно.
**Divergence**: не проблема — никто не пушит в репо пока воркер работает.

### Вариант C: Только увеличить таймаут

`WORKER_SUBPROCESS_TIMEOUT_SECONDS=1500` (25 минут). Уже сделано.

**Плюсы:** тривиально.
**Минусы:** не решает проблему, только снижает вероятность. OOM, сбой сети, перезагрузка — код всё равно потеряется.

## Дополнительно: лимит ретраев PO

Независимо от выбранного варианта, PO нужен лимит на автоматические ретраи engineering task для одного проекта. Сейчас PO увидел failure → сразу тригернул новый task → по кругу. Варианты:

- Счётчик ретраев в task metadata, PO проверяет перед ретраем
- Максимум 2-3 автоматических ретрая, потом — сообщение пользователю "не получается, нужна помощь"
- Экспоненциальный backoff между ретраями

## Подводные камни и Edge Cases (к проработке)

При реализации Варианта B необходимо учесть следующие архитектурные нюансы:

1. **Broken State (Испорченный контекст)**
   - **Проблема**: Предыдущий воркер мог оставить рабочую директорию в невалидном состоянии (завис посреди `git rebase` / `merge conflict`, сломал конфиги так, что проект не собирается).
   - **Решение**: Если воркер фейлится N раз подряд на восстановленном workspace (resume mode), нужен механизм Fallback'а — передавать флаг `force_clean_workspace=true`, чтобы принудительно удалить папку и заставить агента начать с чистого клона.

2. **Concurrency (Состояние гонки)**
   - **Проблема**: Воркеры могут начать писать в одни и те же файлы, если из-за бага PO заспавнит двух engineering-воркеров для одного проекта одновременно.
   - **Решение**: Правило "один воркер на проект одновременно" должно быть железно гарантировано на уровне Worker Manager (необходим mutex/lock по `project_id`).

3. **Синхронизация с удаленным Git (Git Drift)**
   - **Проблема**: Пока воркер был мертв, в remote репозитория могли появиться новые коммиты. Новый воркер стартует со старым локальным workspace, его ветка отстает.
   - **Решение**: В "Resume" промпте явно указать Claude Code проверять состояние репозитория (делать `git status`, `git fetch`), прежде чем слепо продолжать писать код.

4. **Permissions (Права доступа)**
   - **Проблема**: При монтировании папки `/tmp/codegen/workspaces/{project_id}/` в разные контейнеры могут возникнуть проблемы с правами.
   - **Решение**: Убедиться, что `uid/gid` юзера (например, `worker`) внутри контейнера нового воркера полностью совпадают с правами владельца файлов от предыдущего контейнера приложения, чтобы избежать ошибок `Permission denied`.

5. **Stateful-привязка к ноде (Масштабирование)**
   - **Проблема**: Хранение в локальной директории привязывает проект к конкретному хосту, где запущен Worker Manager.
   - **Решение**: Для текущего MVP на одном сервере это абсолютно нормально. Однако при масштабировании на несколько нод (Kubernetes, Docker Swarm) локальная папка работать не будет, и понадобятся shared volumes (NFS, EFS). Нужно зафиксировать этот компромисс.

---

## Решение

**Вариант B** + увеличенный таймаут (вариант C, уже применён) + лимит ретраев PO.

### Изменяемые файлы

| Файл | Изменение |
|------|-----------|
| `services/worker-manager/src/manager.py` | workspace path по `project_id`, skip clone если workspace exists, не удалять workspace в `delete_worker` |
| `services/worker-manager/src/consumer.py` | передавать `project_id` из CreateWorkerCommand |
| `shared/contracts/queues/worker.py` | добавить `project_id` в WorkerConfig |
| `services/langgraph/src/clients/worker_spawner.py` | передавать `project_id` в CreateWorkerCommand |
| `services/langgraph/src/nodes/developer.py` | передавать `project_id`, определять initial vs resume |
| `services/worker-manager/src/workspace.py` | новая функция для GC workspace'ов по возрасту |
| `INSTRUCTIONS.md` (developer prompt) | инструкция про PROGRESS.md |
| `services/langgraph/src/po/tools.py` | лимит ретраев (или в промпте PO) |
