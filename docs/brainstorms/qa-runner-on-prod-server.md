# QA Runner: размещение на прод-сервере

> **Status**: brainstorm
> **Связано с**: [qa-node.md](qa-node.md) (что тестировать), здесь — где и как запускать

## Идея

Вместо QA-воркера в контейнере на стороне оркестратора (как Engineering worker) — запускать Claude Code прямо на продакшен-сервере с проектом пользователя. Он получает playbook, дёргает эндпоинты, тестирует бота через Telethon, браузер через Playwright MCP.

Ключевое отличие от контейнерного воркера: сетевая близость к проекту (localhost), прямой доступ к логам и docker-сети.

## Плюсы

### 1. Нулевая сетевая сложность
- Не нужен туннель/VPN до прод-сервера. Claude уже "на месте" — `curl localhost:8000/health` работает из коробки.
- Для Telegram-ботов — подключение напрямую к compose-сети проекта, доступ к логам контейнеров.

### 2. Максимальная диагностика при фейле
- Прямой доступ к `docker compose logs`, файлам конфигов, `.env`, дампам БД.
- Может сам починить очевидные проблемы (перезапустить контейнер, поправить env).

### 3. Минимум инфраструктуры на стороне оркестратора
- Не нужен новый тип воркера, новый образ Docker, новая очередь. Worker-manager не трогаем.
- Не нужно придумывать как прокинуть сеть между контейнером QA-воркера и удалённым сервером.

### 4. Claude из коробки покрывает 80% кейсов
- `curl`, `httpx`, чтение логов, парсинг HTML — всё уже умеет.
- С Playwright MCP — полное browser testing.
- С Telethon — полное тестирование ботов.
- Бонус "проверь граничные случаи" — реально работает, LLM хорошо генерит edge cases.

### 5. Быстрый старт
- По сути нужно: установить Claude Code на прод-сервер, подготовить скрипт запуска с playbook. Всё.

## Минусы и риски

### 1. Безопасность (главный риск)
- Claude с `--dangerously-skip-permissions` на проде = полный доступ к серверу.
- Prompt injection в ответе тестируемого приложения — Claude может выполнить деструктивные команды.
- Mitigation: непривилегированный юзер, AppArmor/seccomp, но не панацея.

### 2. Blast radius
- В контейнере оркестратора: упал QA-воркер — перезапустили, прод не задет.
- На проде: Claude может `docker compose down` проект, удалить volumes, испортить БД.
- Если на сервере несколько проектов (в будущем) — может задеть соседей.

### 3. Управление жизненным циклом
- Worker-manager контролирует создание/удаление контейнеров, таймауты, liveness checks.
- Claude на проде — как управлять? SSH-сессия? systemd unit? Кто следит что процесс не завис?
- Нужен агент на проде, который принимает команды с оркестратора и запускает/останавливает Claude.

### 4. Обратная связь в оркестратор
- Engineering worker публикует результат в Redis stream — оркестратор знает итог.
- Claude на проде — нужен канал обратной связи:
  - POST на API оркестратора (`/v1/projects/{id}/qa-report`)
  - Файл результата → забирать по SSH
  - Redis напрямую (но прод-сервер вне сети оркестратора)

### 5. Ресурсы прод-сервера
- Claude Code жрёт CPU/RAM, которые нужны пользовательскому проекту.
- Если сервер слабый (1-2 vCPU) — тестирование может тормозить прод.

### 6. Cleanup усугубляется
- В контейнере: всё изолировано, state не протекает.
- На проде: Claude создаёт тестовые данные в продовой БД, нет гарантии что почистит.

## Подводные камни

### Provision агента на проде
Нужно расширить Ansible-плейбук (`prod_infra`):
- Установить Claude Code binary
- Настроить аутентификацию (API key)
- Установить Playwright + браузеры (chromium ~400MB)
- Установить Python + Telethon
- "Лёгкий прод-сервер" превращается в довольно тяжёлую машину.

### Версионирование
Claude Code обновляется. На оркестраторе — один образ, контролируем версию. На N прод-серверах — N копий, нужно обновлять синхронно.

### Аутентификация Claude
- `host_session` (OAuth) — нельзя шарить между серверами
- `api_key` — надо безопасно доставить ключ Anthropic на прод-сервер пользователя

## Компромиссный вариант: контейнер НА прод-сервере

```
Prod Server:
  +-- user-project (compose stack)
  +-- qa-runner (Docker container)
       +-- Claude Code
       +-- Telethon
       +-- Playwright + Chromium
       +-- Подключён к сети user-project
```

- **Изоляция**: контейнер ограничен по ресурсам и правам
- **Сетевой доступ**: подключён к docker network проекта — `curl backend:8000`
- **Нет доступа к хост-системе**
- **Оркестратор управляет** через SSH + `docker run`
- **Результат**: stdout → `docker logs` или POST на API

Даёт сетевую близость (localhost) + изоляцию (container) + управляемость (docker lifecycle).

## Рекомендация

Компромиссный вариант (контейнер на проде) — лучший баланс. Для MVP можно начать с "голого" Claude на проде с жёсткими ограничениями:

1. Отдельный непривилегированный юзер `qa-runner`
2. Только read-access к docker (`docker logs`, `docker inspect`, без `docker rm`)
3. API key authentication
4. Таймаут: `timeout 300 claude ...`
5. Результат: POST на API оркестратора

Потом мигрировать в контейнер, когда формат устаканится.

## Открытые вопросы (специфичные для этого подхода)

1. **Агент на проде** — нужен ли демон на прод-сервере, который слушает команды от оркестратора? Или достаточно SSH + одноразовый запуск?
2. **Образ для контейнерного варианта** — переиспользовать `worker-base-claude` или отдельный `qa-runner` образ с Playwright/Telethon?
3. **Сколько серверов** — пока один, масштабирование не проблема. Когда станет N — нужна стратегия обновления.
