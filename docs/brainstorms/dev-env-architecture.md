# Brainstorm: Ideal Dev Environment Architecture (No DinD)

> **Дата**: 2026-02-19
> **Контекст**: Пользователь запросил переосмысление архитектуры dev-окружения для воркеров. Цель — избавиться от дефолтного DinD (Docker-in-Docker), сделать работу быстрой, ресурсоэффективной, но оставить агенту возможность полноценно использовать Docker (включая volume mounts) по запросу.

---

## Требования к идеальной архитектуре

1. **Полноценный Docker по запросу**: Агент должен мочь поднимать контейнеры, делать volume mounts, при этом пути должны совпадать (без магии с путями хоста внутри контейнера).
2. **On-Demand выделение ресурсов**: Не поднимать тяжеловесный Docker-обвес, если задача этого не требует.
3. **Flat Hierarchy (Без вложенности)**: Контейнеры поднимаются "рядом" с воркером, а не внутри него.
4. **Скорость**: Запуск тестов и скриптов должен быть моментальным (нативная работа предпочтительна).
5. **Универсальность**: То, что работает без Docker, должно работать без Docker (через `uv` / процессы).

---

## Концепция: "Flat Dev Environment" (Плоское окружение)

Уходим от модели "Воркер внутри себя поднимает Docker". Переходим к модели "Воркер — это просто процесс/легкий контейнер в физической сети проекта, а Docker-демон стоит рядом".

### Архитектурные варианты решения задачи полнофункционального Docker без DinD:

### Подход 1: Socket Proxy + Path Translation (Легкий путь, но с костылями)
**Идея**: Оставить проброс `/var/run/docker.sock` в контейнер воркера, но перехватывать вызовы Docker CLI.
- Воркер думает, что делает `docker run -v /workspace:/app`.
- Прокси (или скрипт-обертка над docker CLI) перехватывает путь `/workspace`, понимает, что на хосте это `/tmp/projects/123/workspace`, и подменяет аргумент.
- **Плюсы**: Воркер остается легким, контейнеры поднимаются рядом на хостовом демоне.
- **Минусы**: Хрупко. Сложно перехватить `docker-compose`. Сложно обрабатывать относительные пути.

### Подход 2: Воркер на хосте / Изоляция через Firecracker / Sandbox (Уход от контейнера воркера)
**Идея**: Воркер — это не Docker-контейнер, а защищенный процесс (например, через `bwrap` на Linux или microVM типа Firecracker/Kata).
- Воркер имеет доступ к файлам проекта.
- Воркер имеет доступ к хостовому Docker CLI.
- **Плюсы**: Полностью нативная разработка, мгновенный старт `uv`, 100% совместимость путей (Так как воркер работает почти как скрипт на хосте).
- **Минусы**: Резкое усложнение оркестратора. Нужно менеджить процессы/microVM, а не просто контейнеры. Вопросы безопасности (sandbox escape).

### Подход 3: Sysbox (Надежный Rootless Docker-in-Docker)
**Идея**: Использовать [Sysbox runtime](https://github.com/nestybox/sysbox). Он позволяет запускать systemd и Docker daemon внутри контейнера воркера **нативно** и безопасно (без `--privileged`).
- **Как работает**: Внутри контейнера воркера крутится свой собственный легковесный изолированный `dockerd`.
- **Выполнение требований**:
  - Агент может делать любые volume mounts — они работают, так как демон внутри контейнера.
  - Плоская структура — нет запутанности с путями.
  - Запускать `dockerd` только по запросу: `service docker start` только когда агент решает, что ему это нужно.
- **Минусы**: Sysbox нужно ставить на хосты. Незначительный оверхед на запуск внутреннего dockerd.

### Подход 4: "Orchestrator as a Service" (Толстый клиент)
**Идея**: Забрать права на запуск Docker у агента вовсе. Дать ему Tool (API):
- Агент пишет код.
- Агент хочет проверить: вызывает tool `run_integration_tests(services=["db", "redis"])`.
- Оркестратор (`worker-manager`) на хосте читает `docker-compose.yml` проекта, поднимает нужные базы в сети `dev_123`.
- Оркестратор передает агенту энвы: `DATABASE_URL=postgres://dev_123_db...`
- Агент нативно через `uv` запускает тесты.
- **Плюсы**: Максимальная скорость. Ноль оверхеда, пока не запрошено. Полная изоляция.
- **Минусы**: Агент не может тестировать сложные вещи вроде "соберу-ка я свой Dockerfile и посмотрю работает ли он" без специального тула `build_and_run_image`.

## Предлагаемый гибридный путь (The "Sidecar" approach)

Комбинация **Нативной скорости (uv)** и **On-Demand инфры**.

1. **Воркер — это легкий Python-контейнер.**
   Без Docker демона внутри. Имеет предустановленный `uv`. Все скрипты пишет и гоняет нативно.
2. **Сеть проекта.**
   При создании воркера оркестратор создает Docker-сеть (например, `project_abc_net`). Воркер помещается в эту сеть.
3. **Инфраструктура по запросу (Tooling).**
   Агенту дается набор инструментов (Claude Tools):
   - `request_service(service_name="postgres")` → `worker-manager` поднимает Postgres *в той же сети* и возвращает connection string.
   - `run_compose(file="docker-compose.yml")` → `worker-manager` читает файл с диска (пути на хосте он знает) и поднимает *рядом* с воркером.
4. **Сборка и запуск образов по запросу.**
   Если агенту нужно собрать Dockerfile:
   - Tool `build_and_run(dockerfile="Dockerfile", context=".")`.
   - `worker-manager` на хосте собирает образ, маунтя хостовые пути (он знает маппинг `context="."` -> `/tmp/projects/abc`).
   - Контейнер запускается в сети проекта. Агент может делать к нему HTTP-запросы.
   
### Почему это отвечает требованиям:
- **Вольюмы без ограничений**: Так как сборку и запуск выполняет хостовый `worker-manager`, он сам транслирует пути агента в хостовые пути при mount'ах.
- **Без траты ресурсов**: Базы и сторонние контейнеры не запущены, пока агент не вызовет tool.
- **Без вложенности (рядышком)**: Воркер и тестовая база просто лежат в одной bridge-сети.
- **Быстро**: Код гоняется нативно через `uv`. Докер используется только для баз/инфры.
- **Универсально**: Переносит логику управления Docke-ом из тупого shell-скрипта в типизированные LangGraph Tools.
