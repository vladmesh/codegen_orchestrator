# Brainstorm: Проблемы масштабирования до 15 одновременных клиентов

Отчет о потенциальных узких местах (bottlenecks) и состояниях гонки (race conditions) при одновременной работе 10-15 клиентов без учета лимитов LLM-провайдеров.

## 1. Конфликты (Conflicts & Race Conditions)

*   **GitHub Repository Names Conflict**:
    Если несколько клиентов почти одновременно просят создать проект со схожим функционалом (например, "напиши мне туду апп"), PO может попытаться создать несколько репозиториев с одинаковым или очень похожим именем в рамках одной организации `project-factory-organization`.
    *Результат:* Первый запрос пройдет, а остальные упадут с 422 ошибкой (Repository name already exists).
    *Решение:* На уровне API подмешивать к названию репозитория ID проекта, ID пользователя или короткий хэш (например, `todo-app-a1b2`).

*   **Port Collision при деплое (DEPLOY_PORT)**:
    Если 10 проектов начнут одновременно запускаться на одном и том же `DEPLOY_HOST`, скрипты деплоя могут параллельно просканировать порты и выбрать один и тот же "свободный" `DEPLOY_PORT`.
    *Результат:* `docker-compose up` второго проекта упадет с ошибкой `port is already allocated`.
    *Решение:* Резервация портов должна быть строго атомарной (например, через транзакции в таблице портов базы данных API).

*   **Состояния гонки в Task Dispatcher (Авто-мёрж)**:
    Если 10 агентов одновременно завершают таски для одного проекта, и диспетчер попытается параллельно создать несколько PR (`story/* -> main`), возможны конфликты пушей и мёржей на стороне Git.
    *Требуется:* Убедиться, что логика авто-мёржа строго сериализует операции с гитом для одного репозитория.

## 2. Нагрузка (Load & Resource Exhaustion)

*   **Docker Container Exhaustion / OOM (Worker-Manager)**:
    Каждое сообщение в `engineering:queue` порождает изолированный Docker-контейнер (Claude Code / Factory.ai). 15 клиентов с активными задачами — это 15+ тяжелых контейнеров одновременно (внутри NodeJS, Python + интенсивный I/O).
    *Результат:* Узел `worker-manager` полностью исчерпает RAM (OOM-killer) или намертво ляжет по CPU.
    *Решение:* Жесткий лимит на количество параллельных консьюмеров или внедрение системы управления очередями (пулинг контейнеров) с лимитом параллелизма (например, `max_workers=5`).

*   **GitHub API Secondary Rate Limits**:
    Параллельные операции от 15 агентов (массовые `git push`, API-запросы `set_repository_secrets`, `trigger_workflow_dispatch`, частый поллинг `wait_for_workflow_completion` каждые X секунд) расцениваются как abuse-паттерн.
    *Результат:* GitHub включит защиту от абьюза (secondary rate limits), отдавая 403/429 ошибки на IP-адрес оркестратора.
    *Решение:* Exponential backoff для всех вызовов к GitHub API, агрессивное кэширование и замена поллинга на webhooks.

## 3. Где не хватает параллельности (Sequential Bottlenecks)

*   **PO Consumer (Telegram Queue Polling)**:
    `PO ReactAgent` работает как async consumer. Если консьюмер `po:input` берет сообщения из Redis Stream по одному и дожидается полного ответа агента (вызов LLM + запись стейта в Postgres + генерация ответа) перед чтением следующего, то сообщения 15 клиентов выстроятся в строгую общую очередь.
    *Результат:* Если один юзер задал сложный архитектурный вопрос (требующий цепочки тулов), все остальные будут ждать ответа бота минуты.
    *Решение:* PO Consumer должен реализовать concurrency (worker pool) — обрабатывать сообщения от *разных* `user_id` параллельно, блокируя выполнение только внутри одной сессии.

*   **Task Dispatcher**:
    Периодический пуллинг в базу раз в 30 секунд.
    *Результат:* Под нагрузкой процесс обнаружения завершенных тасок и старта новых Runs вызовет искусственную задержку всего пайплайна, превратив его в batch-сессию.
    *Решение:* Ускорить поллер или перейти на event-driven модель (реагировать сразу по факту прилета события о статусе).

*   **Синхронное ожидание (Wait for Workflow Completion в Deployer)**:
    Deployer ждёт завершения `deploy.yml` в GitHub Actions через API поллинг с таймаутом в 600 секунд.
    *Результат:* Если это честный цикл `while/sleep` внутри потока/воркера LangGraph, то 15 параллельных деплоев полностью исчерпают пул активных потоков воркера, застопорив всю очередь деплоя.
