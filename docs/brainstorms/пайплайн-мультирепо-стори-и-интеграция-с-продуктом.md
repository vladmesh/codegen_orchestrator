# Future Pipeline Improvements & Open Ideas

Этот документ содержит оставшиеся концепции и архитектурные наброски по улучшению dev-процесса и самого оркестратора, которые пока не реализованы.

## 1. Мультирепо и Repository Entity
Текущая модель "опубликуй проект в один репозиторий" ломается, когда нужно коммитить и в код `codegen_orchestrator`, и в `service-template`.
- **Идея**: Внедрить агрегирующую сущность `Repository` в БД.
- **Связи**: `Task` должен зависеть от (Foreign Key) `Repository`.
- **Поведение CLI агентов**: `/next` и `/implement` будут использовать `repository_id` для автоматического переключения Git-контекста (соответствующей директории или воркера), не смешивая контексты разных проектов.

## 2. Интеграция User Stories
Вместо текстового файла `USER_STORIES.md` нужные четко измеримые бизнес-сущности.
- **Идея**: Сущность `Story` в БД. У стори могут быть вложенные `Task` (бывшие `work_items`).
- **Бенефиты**: Прозрачность прогресса (стори закрыта, когда закрыты все её таски).
- **Epic**: Верхнеуровневая сущность для группировки Story (`parent_story_id`).

## 3. Умная декомпозиция (Architect Node vs Plan Node)
С появлением Story нам нужен скилл, который конвертирует бизнес-фичу в набор технических задач для разных репозиториев (мультирепо).
- **Скилл `/architect` (продуктовый)**: Читает `Story` и `Brainstorm` (отчеты), разворачивает их в набор `Task`, проставляет связи `depends_on` и назначает им целевые репозитории (БД `work_items`).
- **Скилл `/plan` (инженерный)**: Остаётся для предварительного ресёрча в рамках одной конкретной технической `Task` *перед* началом кодинга.

## 4. Tester Node / E2E API
Наша валидация E2E сейчас просто выводится в консоль/Discord.
- **Идея**: Автоматизировать сбор метрик, настроив скилл `/e2e-run` на создание записей (например, `QA_Run` или `verification_events`) в рамках истории задачи. Это даст четкую статистику фейлов и сброса задач в `/triage` для багфиксов.

## 5. Human-in-the-loop (Admin Spike Task)
Автоматизация декомпозиции (Архитектор) может забуксовать на очень сложных архитектурных задачах.
- **Идея**: Внедрить fallback на `Spike Task`. Админ может подключиться в сессию чата, в ручном режиме вместе с LLM составить brainstorm/декомпозицию, и после завершения этого "спайка" вернуть проект в автоматический цикл (`unfreeze`).
- Смысл в том, чтобы система ожидала таких полуручных (human-in-the-loop) шагов как нативную часть процесса.

## 6. Jira-style Comment Log
В таблице `work_item_events` уже заложен тип `COMMENT` для трекинга обсуждений.
- **План**: Пока откладываем UI-реализацию для пользователей в Телеграме. Обсуждения продуктовых и технических деталей задач станут удобными только с появлением веб-дашборда оркестратора.
