# Typed environment contract: MVP migration

## Status

Slice 3 requires the contract for every deploy. `codegen_orchestrator-607` completed the Slice 4 removal of environment classification: deployment no longer has an analyzer fallback or an LLM path.

## Problem

Deploy currently reads every key from `.env.example` and classifies it as `infra`, `computed`, or `user` using name tables, patterns, and an LLM fallback. This mixes local, test, and production settings and treats public configuration, allocations, generated credentials, and user credentials as one `resolved_secrets` bag.

The Stage 7 run on `service-template 0.3.1` demonstrated the failure mode: `POSTGRES_HOST_PORT` and `REDIS_HOST_PORT` are local-only Compose settings, but the analyzer classified them as user secrets and stopped production deploy even though production Compose does not publish those ports.

The complete env set cannot be frozen at scaffold time. Every engineering story may add variables. The contract therefore has to evolve with the code and be validated at each story merge.

## Target model

The source of truth is an explicit contract committed with the generated project. The template supplies the baseline; every worker that adds or changes environment consumption updates the contract in the same PR. CI checks the declarations against actual usage. Deploy consumes the contract and does not infer production semantics from `.env.example`.

Use per-owner fragments to reduce conflicts:

```text
infra/env.contract.yaml
services/backend/env.contract.yaml
services/tg_bot/env.contract.yaml
services/frontend/env.contract.yaml
```

Minimal entry:

```yaml
STRIPE_SECRET_KEY:
  source: user_secret
  environments: [production]
  consumers: [backend]
  required: true
  description: Stripe server API key

BACKEND_PORT:
  source: allocation
  environments: [production]
  consumers: [backend]
  service: backend
  exposure: public

POSTGRES_HOST_PORT:
  source: literal
  environments: [local]
  value: 5432
```

MVP sources:

- `user_secret`: value must be supplied by a user and stored encrypted at project scope.
- `generated_secret`: value is generated by the platform, stored encrypted, and reused on redeploy.
- `allocation`: value comes from the allocation API for a named service/resource.
- `derived`: non-secret value computed from trusted project, repository, registry, or server context.
- `literal`: non-secret value committed in the contract, normally scoped to local/test.

Do not keep `infra` as a source type. It currently conflates credentials, URLs, database names, service discovery, and ports.

## Lifecycle

1. `service-template` creates baseline fragments for selected modules.
2. Architect records expected external dependencies so PO can request user credentials early. This is planning input, not the final contract.
3. Engineering workers must update the relevant fragment whenever their code introduces, removes, or changes an env variable.
4. CI extracts observable usage from Python settings/accesses, Compose interpolation, workflows, and shell entrypoints, then compares it with the fragments.
5. CI merges fragments into a canonical artifact bound to the tested commit SHA.
6. Deploy selects only entries containing `production`, resolves each typed source, and fails with a typed outcome if resolution is impossible.
7. A later story may extend the contract again; there is no one-time finalization step.

## MVP gates

CI must fail when:

- a statically observable env key is used but not declared;
- two fragments declare the same key incompatibly;
- a production entry has an unknown source or resolver;
- a `user_secret` has no consumer or description;
- an `allocation` lacks a service/resource selector;
- a sensitive value is declared as `literal` or `derived`;
- the canonical artifact cannot be reproduced from the checked-in fragments.

Declared-but-not-observed keys are warnings in the MVP because dynamic settings access exists. They can become errors after the scanner matures.

Shell entrypoints are also warnings in the MVP. Their scanner is deliberately
limited to shell files and shell shebangs, but is not yet a syntax-complete
shell parser. An undeclared static shell reference is reported in CI without
blocking artifact publication; it becomes an error when the scanner matures.

Workflow references are observable deployment wiring, not runtime entries of a
generated project's contract. CI records them for audit, but excludes them from
the undeclared-key and required-but-not-observed gates. The orchestrator
supplies deployment credentials such as DOTENV, REGISTRY_*, and DEPLOY_*.

## Runtime outcomes

Replace the generic missing-secret branch with distinct outcomes:

- `waiting_for_user_secret`
- `allocation_missing`
- `environment_contract_invalid`
- `environment_resolution_failed`
- `deployment_failed`

Generated/user secrets, computed env, and allocations remain separate typed maps until the final dotenv assembly. Only secret maps are persisted in `Project.config.secrets`.

## Role of agents

An LLM is not authoritative in the production deploy path.

- Architect proposes expected external credentials from product requirements.
- Engineering worker owns contract changes introduced by its code.
- Deterministic CI proves declaration coverage and schema validity.
- An env-audit agent is optional for legacy imports, large refactors, and ambiguous findings. It proposes a patch; it does not choose runtime values during deploy.

## Non-goals for MVP

- A general-purpose secrets manager or user vault.
- Automatic rotation of all provider credentials.
- Perfect static extraction of dynamic env names.
- Kubernetes or multi-cloud secret injection.
- Replacing the current GitHub Actions deploy transport in the same migration.
